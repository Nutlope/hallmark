why 18ms matters

Slow ingestion is why you don't trust your error tracker.

When your tracker takes 800ms to acknowledge a payload, you start dropping events at the edge to protect throughput. Then you wonder why your 2am page didn't fire. The fix is not a bigger queue. The fix is ingestion that finishes before the request does.

supported runtimes

Three runtimes, one agent, one wire format.

Node

>= 18.0

# package.json
npm install @flarepin/node

# index.ts
import flarepin from "@flarepin/node";
flarepin.init({ project: "checkout", env: "prod" });
  • uncaught + unhandled rejections

deploy

Run it yourself, or let us run it.

Both paths use the same agent and the same wire format. You can start managed, export your data, and self-host later. No lock-in, no different SDK, no migration script.

self-hosted

apache-2.0 · single binary + postgres

# docker-compose.yml
services:
  flarepin:
    image: flarepin/server:2.4
    ports: ["8080:8080"]
    environment:
      DATABASE_URL: postgres://flarepin@db/flarepin
      INGEST_KEY: $INGEST_KEY
  db:
    image: postgres:16
    volumes: ["pgdata:/var/lib/postgresql/data"]

volumes: { pgdata: {} }
  • your hardware, your data, your retention policy
  • no outbound traffic from the agent unless you point it out
  • upgrade by pulling a tag, no migration scripts to run by hand
